# 01. 내부 기술심사원 운영가이드

> **내부 업무용 / 대외 공개 금지**

## 1. 목적과 적용범위

본 가이드는 기업이 제출한 AI 제품·서비스 신청자료를 바탕으로 다음을 일관되게 판단하기 위한 내부 기준이다.

1. 제품·서비스에 인공지능처리 연산체계가 실제 적용되어 있는가.
2. 해당 연산체계가 입력을 처리하여 예측·분류·생성·탐지·최적화 등 유의미한 출력을 만드는가.
3. AI가 제품의 주요 기능과 구조적으로 결합되어 있는가.
4. 신청한 주요 AI 기능이 실제 운영환경에서 정상 동작하는가.
5. 외부 AI 서비스를 사용한 경우 실제 호출·연동 관계가 확인되는가.

심사원은 신청인의 기술수준이나 AI 성능의 우수성을 평가하는 것이 아니라, **AI 적용 사실과 제품 기능과의 결합 관계를 객관적으로 확인**한다.

## 2. 심사 기본원칙

### 2.1 주장-근거-재현의 3단 연결

모든 핵심 판단은 다음 세 요소가 연결되어야 한다.

- **주장:** 신청서 또는 기능명세에 기재한 내용
- **근거:** 구조도, 모델 정보, 코드·설정, 로그, 파일, 시험결과 등
- **재현:** 현장 시연 또는 동일 조건에서 다시 확인 가능한 동작

세 요소 중 하나가 없다고 즉시 부적합으로 판단하지는 않지만, 핵심 항목을 판단할 수 없을 정도이면 보완이 필요하다.

### 2.2 명칭보다 동작을 확인

‘AI’, ‘딥러닝’, ‘LLM’, ‘지능형’이라는 표현 자체는 증거가 아니다. 다음을 확인한다.

- 어떤 입력이 어느 모듈로 전달되는지
- 모듈이 어떤 모델·알고리즘·외부 API를 사용하는지
- 어떤 출력이 생성되는지
- 출력이 제품의 어떤 기능에 사용되는지

### 2.3 기능 단위 심사

제품 전체를 포괄적으로 ‘AI 제품’이라 부르는 방식보다, 신청한 핵심 AI 기능을 기능번호별로 분리하여 확인한다.

- 하나의 기능에 서로 다른 모델이 직렬·병렬로 결합되면 E 혼합형으로 볼 수 있다.
- 서로 독립된 AI 기능은 기능번호를 나누어 각각 구현방식을 선택한다.
- 동일 모델을 여러 화면에서 재사용하더라도 입력·출력·사용 목적이 다르면 별도 기능으로 분리할 수 있다.

### 2.4 최소 공개 원칙

심사에 필요한 범위만 확인한다.

- API Key 전체 문자열은 수집하지 않고 앞·뒤 일부를 마스킹한다.
- 개인정보 원천데이터 대신 비식별 샘플 또는 메타데이터를 확인한다.
- 모델 가중치 원본 제출이 곤란하면 해시, 파일목록, 로딩 로그, 배포 산출물 등 대체증빙을 인정할 수 있다.
- 영업비밀 코드는 전체 제출보다 관련 함수·설정·실행 로그의 제한적 열람을 우선한다.

- 구조도·명세서·기타 참고자료의 파일 접근 가능 여부 확인
- 구현방식 선택과 조건부 필드 입력이 일치하는지 확인
- 동일 기능이 중복 등록되거나 서로 다른 기능이 한 행에 혼재하지 않았는지 확인

### 단계 2. 제품 범위 확정

다음 질문에 답할 수 있어야 한다.

- 심사대상 제품·서비스의 경계는 어디까지인가.
- 신청한 제품 버전 또는 모델은 무엇인가.
- 외부 시스템이나 고객사 환경에 의존하는 부분은 무엇인가.
- AI 모듈이 실행되는 위치는 클라우드, 사내 서버, 단말, 장치 중 어디인가.

제품 범위가 불명확하면 동일 시연이라도 다른 제품의 기능을 보여줄 위험이 있다.

### 단계 3. 기능명세 필드 검토

각 기능에 대해 다음 순서로 읽는다.

1. 기능명과 인공지능역할로 업무 목적을 파악한다.
2. 입력과 출력으로 AI 처리의 시작·종료점을 확인한다.
3. 구현방식과 전용 필드로 실제 AI 엔진을 식별한다.
4. 연산자원·실행환경으로 실행 가능성을 검토한다.
5. 상세 입력·출력 설명으로 데이터 흐름을 구체화한다.
6. 레퍼런스참조위치와 참고자료로 주장을 문서 증거에 연결한다.

### 단계 4. 서류 확인기준 판정

- 1.1 인공지능처리 연산체계 적용 여부
- 1.2 학습·추론 등 AI 특성
- 1.3 입력-연산-출력의 유의미성
- 2.1 구조도상 위치와 제품 구동체계 결합
- 2.2 기능·편의성·접근성·효율성 활용
- 3.1 외부 AI 서비스 연동 증빙

2.3은 원칙적으로 현장 동작 확인이 필요하다.

### 단계 5. 현장 확인 시나리오 설계

심사원이 임의로 테스트를 즉석에서 만들기보다, 사전 검토한 기능명세를 기준으로 다음 세 종류를 준비한다.

- 정상 입력: 기업이 예상한 대표 사용 시나리오
- 변형 입력: 값·강도·표현·형식이 달라진 입력
- 경계·예외 입력: 미지원 형식, 낮은 품질, 범위 밖 값, 네트워크 단절 등

### 단계 6. 현장 시연 및 증적 확보

기능번호별로 다음 순서를 기록한다.

1. 제품명·버전·실행환경 확인
2. 입력 데이터 제시
3. AI 기능 실행
4. 모델/API 호출 또는 연산 흔적 확인
5. 출력 결과 확인
6. 제품 후속 동작 확인
7. 캡처·로그·영상 파일명 기록

### 단계 7. 보완 여부 결정

보완은 ‘기술이 없다고 판단’한 상태가 아니라, **현재 자료로는 확인할 수 없으나 추가 자료로 판단 가능성이 있는 상태**다.

보완 요청은 다음 네 요소를 포함한다.

- 현재 제출 내용
- 확인되지 않는 사항
- 필요한 자료 또는 시연
- 해당 확인기준 번호

### 단계 8. 종합판정 및 보고서 작성

항목별 판단의 논리를 먼저 작성한 후 종합판정을 입력한다. 종합의견에는 다음이 포함되어야 한다.

- 확인한 핵심 AI 기능 수와 명칭
- 적용된 AI 연산체계 또는 외부 서비스
- 제품 기능과의 결합 관계
- 현장 시연 결과
- 미비사항 또는 제한사항
- 종합판정의 근거

## 4. 항목별 상태 정의

| 상태 | 사용 조건 | 예시 |
|---|---|---|
| 적합 | 제출자료와 시연으로 해당 기준이 객관적으로 확인됨 | 모델명·실행로그·입출력 결과가 일치 |
| 보완 필요 | AI 적용 가능성은 있으나 자료 누락·불일치로 현재 판단 불가 | Base Model은 적었으나 실제 적용 버전과 배포 산출물 미확인 |
| 부적합 | 기준을 충족하지 않음이 확인되거나 핵심 사실이 신청내용과 다름 | 규칙 기반 임계치 계산을 AI 예측으로 신청 |
| 해당 없음 | 기준의 전제가 존재하지 않음 | 외부 API를 전혀 사용하지 않는 A 유형의 3.1 |

‘해당 없음’은 통과와 다르다. 종합판정 시 적용대상 기준에서 제외한다.

## 5. 증빙의 우선순위

### 5.1 직접증빙

- 모델 파일 및 해시
- 모델 로딩 로그
- 학습·튜닝 실행 이력
- 추론 호출 로그
- 외부 API 요청·응답 메타데이터
- GPU/NPU/CPU 사용량 변화
- 구조도와 실제 배포 설정의 일치
- 현장 시연 결과

### 5.2 보조증빙

- 개발자 설명
- 제안서·홍보자료
- 화면 캡처만 있는 매뉴얼
- 성능 수치만 있는 시험성적서

보조증빙만으로 핵심 사실을 확정하지 않는다. 예를 들어 정확도 95%라는 문구는 AI 모델의 존재를 간접적으로 시사하지만, 어떤 모델이 제품에 탑재되었는지는 별도로 확인해야 한다.

## 6. 기능 중요도 판단

AI 기능이 제품의 주요 기능과 결합되었는지 판단할 때 다음을 종합한다.

- 제품 설명과 조달·판매 단위에서 해당 기능이 핵심 가치로 제시되는가.
- AI 모듈을 제거하면 기능 수행이 불가능하거나 품질·속도·접근성이 실질적으로 저하되는가.
- AI 출력이 제품의 후속 처리·판정·제어·사용자 의사결정에 사용되는가.
- 단순 도움말, FAQ, 범용 챗봇, 외부 화면 링크 수준에 그치지 않는가.
- AI가 제품 데이터·업무 규칙·워크플로와 기술적으로 결합되어 있는가.

## 7. 대표적인 위험신호

- ‘자체 AI’라고 기재했으나 실제로는 외부 API만 호출
- ‘학습’이라고 표현했으나 사용자 설정값 저장 또는 통계 누적에 불과
- 모델명 없이 ‘당사 독자 알고리즘’만 기재
- 입력·출력이 제품 화면 기준으로만 작성되어 AI 모듈의 I/O가 보이지 않음
- 구조도에 AI 모듈은 있으나 실제 배포환경에는 모델 파일이나 호출설정이 없음
- 데모 영상은 있으나 제품 버전·시점·환경을 확인할 수 없음
- 외부 API 응답을 그대로 표시하고 제품 내부 처리·기능 결합이 없음
- 고정 규칙, 임계치, 키워드 매칭을 AI 판단으로 표현

## 8. 심사원 기록 규칙

- 사실과 판단을 분리한다.
  - 사실: “GPU 사용률이 기능 실행 전 4%, 실행 중 63%로 변동됨.”
  - 판단: “추론 시점과 연산자원 사용 증가가 일치하여 모델 실행 흔적이 확인됨.”
- 확인하지 못한 사실을 추정형 문장으로 채우지 않는다.
- 기업 설명은 “신청인 설명에 따르면”으로 구분한다.
- 파일명·화면명·로그시각·모델버전을 가능한 한 구체적으로 남긴다.
- 동일 항목의 판단이 기능마다 다르면 기능번호별로 나누어 기재한다.

## 9. 권장 증적 파일명

증적자료는 접수번호별 폴더에 저장하므로 파일명에는 보고서의 세부 ID만 사용한다.

`[세부 ID].[확장자]`

예시:

- `F01-01.jpg`
- `F01-05.png`
- `F02-04.pdf`

## 10. 최종 자체점검

- [ ] 모든 기능명세 행을 검토했다.
- [ ] 구현방식별 필수 필드를 확인했다.
- [ ] 구조도와 실제 실행환경이 일치한다.
- [ ] 정상·변형·예외 입력을 확인했다.
- [ ] 외부 API 사용 기능의 로그 또는 설정을 확인했다.
- [ ] 민감정보가 증적에 노출되지 않았다.
- [ ] 보완요청은 항목 번호와 필요한 자료가 명확하다.
- [ ] 종합의견이 항목별 판단과 모순되지 않는다.

# 02. 기능명세 필드별 심사가이드

> **목적:** 기업 신청 입력폼의 기능명세 필드를 심사원이 동일한 수준으로 해석하도록 필드별 작성 기대수준, 확인방법, 증빙, 보완·부적합 경계를 정의한다.

## 1. 필드 구조 요약

### 공통 요약 필드

`기능번호` · `기능명` · `인공지능역할` · `입력` · `출력` · `레퍼런스참조위치`

### 구현 공통 필드

`구현방식` · `연산자원요약` · `실행환경요약` · `입력데이터설명` · `출력데이터설명` · `기타참고자료파일명`

### 구현방식별 필드

- A: `학습데이터사양`, `개발환경라이브러리알고리즘`
- B: `BaseModel명칭`, `튜닝방법`, `튜닝데이터셋`
- C: `외부API정보`
- D: `타겟HW_OS`, `추론런타임`
- E: `혼합구성설명`, `모델별역할및입출력흐름`

## 2. 공통 요약 필드

### 2.1 기능번호

| 항목 | 내용 |
|---|---|
| 작성 목적 | 기능별 신청정보·현장 시나리오·증적을 연결하는 고유 식별자 |
| 기대 수준 | 한 접수 건 안에서 중복되지 않는 번호. `1`, `2`, `3` 또는 `F01`, `F02` 등 일관된 규칙 |
| 심사 확인 | 기능 수와 행 수가 일치하는지, 증적 파일명과 연결 가능한지 |
| 적정 예 | `F01` |
| 미흡 예 | 모든 기능을 `1`로 기재, 기능번호 공란 |
| 조치 | 단순 표기 누락이면 보완. 기능 간 경계가 불명확하면 기능 재분리 요청 |
| 관련 기준 | 전체 추적성, 특히 2.3 |

### 2.2 기능명

| 항목 | 내용 |
|---|---|
| 작성 목적 | 사용자가 인식하는 제품 기능 단위와 AI 처리 단위를 식별 |
| 기대 수준 | 동작과 결과가 드러나는 명칭. ‘AI 기능’, ‘분석’, ‘플랫폼’처럼 범용적인 명칭은 지양 |
| 심사 확인 | 제품 메뉴·매뉴얼·구조도·시연 화면의 명칭과 동일하거나 대응관계가 설명되는지 |
| 적정 예 | `생산품 외관 결함 자동 탐지` |
| 미흡 예 | `인공지능`, `스마트 기능`, `데이터 처리` |
| 보완 기준 | 명칭만으로 기능 경계를 알 수 없거나 여러 독립 기능을 한 행에 묶음 |
| 부적합 위험 | 기재한 기능이 제품에 존재하지 않음 |
| 관련 기준 | 1.1, 2.2, 2.3 |

### 2.3 인공지능역할

| 항목 | 내용 |
|---|---|
| 작성 목적 | 해당 기능에서 AI가 수행하는 구체적 연산·판단 역할을 정의 |
| 기대 수준 | ‘무엇을 입력받아 어떤 추론·생성·분류·탐지·예측을 하는지’가 한두 문장에 나타나야 함 |
| 심사 확인 | 단순 데이터 조회·정렬·규칙계산과 구분되는지, 출력이 제품 후속 기능에 사용되는지 |
| 적정 예 | `카메라 영상의 객체를 탐지하고 안전모 미착용 작업자를 분류하여 경보 후보를 생성한다.` |
| 미흡 예 | `업무를 자동화한다.`, `사용자 편의를 높인다.` |
| 보완 기준 | AI 역할과 제품 효과만 혼재하여 실제 연산이 보이지 않음 |
| 부적합 위험 | 실제 동작이 임계치 비교·키워드 매칭뿐임 |
| 관련 기준 | 1.1, 1.2, 1.3, 2.2 |

### 2.4 입력

| 항목 | 내용 |
|---|---|
| 작성 목적 | AI 처리의 시작점과 입력 유형을 짧게 요약 |
| 기대 수준 | 데이터 유형+주요 출처. 상세 스키마는 `입력데이터설명`에 작성 |
| 적정 예 | `산업용 카메라 RGB 이미지, 생산품 ID` |
| 미흡 예 | `데이터`, `사용자 입력`, `DB` |
| 심사 확인 | 시연 시 실제 입력과 일치하는지, 제품 입력과 AI 모듈 입력이 다르면 변환관계가 설명되는지 |
| 관련 기준 | 1.3, 2.3 |

### 2.5 출력

| 관련 기준 | 1.3, 2.2, 2.3 |

### 2.6 레퍼런스참조위치

| 항목 | 내용 |

## 2. 항목 1.1 — 인공지능처리 연산체계 적용

### 서류 확인

- 기능명세의 기능명·AI 역할·구현방식
- 모델명·라이브러리·외부 API·추론런타임

### 적합

- 외부 AI 앱 링크 또는 화면 표시만 제공
- 기재한 모델·API가 실제 제품에서 사용되지 않음

## 3. 항목 1.2 — 학습·추론 등 AI 특성

적용된 연산체계가 단순 규칙 실행이 아니라 학습된 파라미터 또는 추론 메커니즘을 사용하여 입력에 대한 결과를 생성하는가.

### 구현방식별 확인

| 방식 | 주요 확인자료 | 현장 핵심 |
|---|---|---|
| A | 학습데이터, 학습코드·로그, 모델 가중치, 검증결과 | 모델 로딩, 미학습 입력 추론, 학습 산출물 연결 |

| D | 모델 엔진 파일, 런타임, 장치 로그 | 네트워크 차단 상태 로컬 추론, 자원사용 |
| E | 구성요소별 A~D 증빙 | 각 모델의 추론과 결합 흐름을 모두 확인 |

### 적합

다음 중 하나 이상이 객관적으로 확인된다.

- 학습으로 생성된 모델 파라미터·체크포인트
- 사전학습 모델을 실제 추론에 사용
- 외부 AI 서비스가 실시간 추론·생성을 수행
- 미학습 신규 입력에 대해 모델 결과를 생성
- 모델 실행 시 추론 로그 또는 연산자원 사용이 확인

### 보완 필요

- 모델명은 있으나 실제 로딩 파일·버전이 불명확
- 학습을 주장하나 데이터·로그·산출물이 전혀 없음
- 온디바이스라고 기재했으나 로컬·클라우드 실행 위치 불명확

### 부적합

- 학습·추론이 아닌 고정 수식·조건문·키워드 매칭만 사용
- 미리 저장한 결과를 입력과 무관하게 출력
- AI 모델이 존재하더라도 신청 기능 실행 시 호출되지 않음

### 현장 테스트

1. 신규 입력 준비
2. 기능 실행 전 모델 프로세스·자원상태 확인
3. 기능 실행
4. 모델 호출·자원변동·로그 확인
5. 입력 변형 후 결과 변화 확인

## 4. 항목 1.3 — 유의미한 결과 출력

### 판정 질문

제품에 입력된 데이터가 AI 연산체계를 거쳐 정상 범위의 결과로 출력되며, 입력 변화와 출력 변화 사이에 합리적인 관계가 있는가.

### 서류 확인

- 입력·출력 및 상세 설명
- 라벨·값 범위·신뢰도 정의
- 후처리·임계치
- 시험결과 또는 모델카드
- 예외처리 방식

### 현장 시험 유형

#### 4.1 반복 입력

- 결정론적 모델: 동일 입력에 동일 또는 허용오차 내 결과
- 생성형·확률적 모델: 내용이 달라도 목적과 형식에 부합하는 합리적 결과

#### 4.2 변형 입력

- 밝기·각도·노이즈·문장표현·수치 등 하나의 요인을 단계적으로 변경
- 출력 점수·라벨·생성내용이 합리적으로 반응하는지 확인

#### 4.3 미학습·신규 입력

- 신규 샘플에서 결과 생성 여부
- 낮은 신뢰도, 미지원 표시, Fallback 등 안전한 처리

#### 4.4 잘못된 입력

- 빈 파일, 손상 파일, 미지원 형식, 범위 밖 수치
- 오류 메시지와 시스템 안정성 확인

### 적합

- 입력이 실제 모델에 전달되고
- 결과가 정의된 형식·범위로 출력되며
- 대표·변형 입력에 합리적으로 반응함

### 보완 필요

- 대표 입력 하나만 시연하여 입력-출력 관계 판단이 어려움
- 출력 의미·값 범위가 정의되지 않음
- 시험데이터가 모두 학습데이터와 동일함

### 부적합

- 결과가 입력과 무관하게 고정됨
- 모델 오류로 정상 출력이 불가능
- 기본적인 예측·생성 품질이 신청 기능 수행이 어려울 정도로 붕괴
- AI 결과라고 주장한 값이 실제로는 규칙 계산값

## 5. 항목 2.1 — 구조도상 위치와 제품 결합

### 판정 질문

AI 모듈의 실행 위치, 입력·출력 데이터 흐름, 제품의 다른 구성요소와의 연결이 구조도와 실제 환경에서 명확한가.

### 구조도 필수 요소

- 사용자 또는 외부 시스템 입력
- 전처리 모듈
- AI 모델/API/런타임 명칭
- 모델 실행 위치와 네트워크 경계
- 후처리·업무로직
- DB·파일·메시지큐
- 사용자 화면·제어장치 등 최종 기능
- 외부 서비스와 데이터 전송 방향

### 적합

- 구조도와 배포환경이 일치
- AI 모듈이 실제 제품 흐름 안에 존재
- AI 출력이 후속 기능으로 전달됨

### 보완 필요

- 구조도에 ‘AI 서버’만 있고 모델·입출력·위치가 없음
- 외부 API와 사내 모듈의 경계 불명확
- 신청기능의 흐름이 구조도에서 식별되지 않음

### 부적합

- AI 모듈이 제품과 독립된 별도 데모
- AI 제거 후에도 신청한 주요 기능이 동일하게 동작
- 구조도와 실제 배포구조가 본질적으로 다름

### 제거 영향 확인

AI 모듈을 실제 삭제할 필요는 없다. 다음 대체방법을 사용할 수 있다.

- 모델 프로세스 중지
- API 호출 차단 또는 테스트 키 비활성화
- 모델 결과 대신 오류 반환
- 기능 플래그 비활성화

그 결과 주요 기능이 중단·저하되는지 확인한다.

## 7. 항목 2.3 — 주요 기능 정상 동작

### 판정 질문

신청한 각 핵심 AI 기능이 대상 제품 버전과 운영환경에서 정상적으로 수행되는가.

### 기능별 최소 시연

- 대표 정상 시나리오 1건 이상
- 변형 입력 시나리오 1건 이상
- 예외 또는 경계 시나리오 1건 이상
- 결과가 제품 후속 기능에 반영되는 과정

### 적합

- 주요 경로에서 오류 없이 동작
- 입력과 결과가 기능명세와 일치
- 제품 화면·저장·제어 등 후속 처리가 정상

### 보완 필요

- 개발환경 데모만 있고 신청 제품 버전에서는 미확인
- 일부 기능은 동작하나 중요 기능이 시연되지 않음
- 외부 장애로 당일 확인하지 못했으나 대체증빙 가능

### 부적합

- 주요 기능에 재현 가능한 결함이 존재
- 입력을 처리하지 못하거나 결과가 기능 목적에 맞지 않음
- 신청한 버전에는 해당 기능이 배포되지 않음

### 결함과 부적합의 구분

사소한 UI 오탈자나 비핵심 표시 문제는 즉시 부적합 사유가 아니다. 다만 AI 입력 전달, 추론 실행, 결과 생성, 후속 기능 중 하나가 실패하면 핵심 기능 결함으로 본다.

## 8. 항목 3.1 — 외부 AI 서비스 연동

### 적용대상

C 유형과 외부 API가 포함된 E 유형. A·B·D라도 일부 기능에서 외부 서비스를 호출하면 해당 부분은 적용한다.

### 확인자료

- 외부 서비스 제공자·모델·배포명
- API endpoint 또는 SDK 설정
- 인증정보의 존재(마스킹)
- 호출시각·request ID·응답코드·토큰/사용량 로그
- 청구·사용량 대시보드
- 제품 내부 호출 모듈과 오류처리

### 적합

- 기능 실행시점에 실제 외부 호출이 발생하고
- 설정한 모델·배포와 신청내용이 일치하며
- 응답이 제품 기능에 반영됨

### 보완 필요

- API Key 존재만 확인되고 호출 로그가 없음
- 호출 로그는 있으나 신청기능과 request ID 연결이 불명확
- 로그 보존정책으로 원본을 제시하기 어려우나 대체 대시보드 가능

### 부적합

- 실시간 호출이 발생하지 않음
- 다른 제품 또는 개발테스트의 로그를 제시
- 외부 화면 링크만 제공
- 기재 모델과 실제 호출 모델이 다름

### 보안 주의

보고서에는 API Key 전체값을 기록하지 않는다. 예: `sk-proj-12****9X`처럼 마스킹하고 Secret Manager 경로·설정 화면·호출 성공로그로 존재를 확인한다.

## 9. 현장심사 표준 시나리오 템플릿

| 순서 | 확인내용 | 심사원 기록 |

# 05. 보완요청 및 심사의견 문구

> 문구는 사실관계와 확인기준 번호를 맞춰 수정하여 사용한다. 신청인에 대한 공식 통지 권한과 절차는 확인기관의 운영기준을 따른다.

## 1. 보완요청 작성 원칙

좋은 보완요청은 다음 네 가지가 명확하다.

1. 어떤 신청내용을 검토했는가.
2. 무엇이 확인되지 않았는가.
3. 어떤 자료 또는 시연이 필요한가.
4. 어떤 확인기준과 연결되는가.

피해야 할 표현:

- `AI 증빙을 제출하세요.`
- `내용이 부족합니다.`
- `학습 여부를 확인할 수 없습니다.`

권장 표현:

- `기능 F01의 구현방식이 A(자체 학습 모델)로 기재되어 있으나, 학습데이터의 출처·규모·라벨 정의와 실제 배포 모델 산출물의 연결정보가 없어 확인기준 1.2 판단이 어렵습니다. 학습데이터 사양, 학습 또는 모델 생성 로그, 운영환경에서 로딩되는 모델 파일명·해시를 보완해 주시기 바랍니다.`

## 2. 공통 필드 보완 문구

### 기능명·AI 역할

`기능 F__의 기능명과 인공지능 역할이 ‘AI 분석’으로만 기재되어 있어 AI가 수행하는 구체적 연산과 제품 기능을 식별하기 어렵습니다. 입력 데이터, AI의 처리 역할(예측·분류·생성·탐지 등), 생성되는 결과를 구분하여 보완해 주시기 바랍니다. [관련 기준 1.1, 1.3, 2.2]`

### 입력·출력

`기능 F__의 입력·출력이 각각 ‘데이터’, ‘분석결과’로 기재되어 있어 입력-연산-출력 흐름을 확인하기 어렵습니다. 실제 AI 모듈에 전달되는 데이터 유형·주요 필드·전처리와 모델이 직접 반환하는 라벨·점수·텍스트 등의 결과 형식을 보완해 주시기 바랍니다. [관련 기준 1.3]`

### 레퍼런스 참조위치

`기능 F__의 참고문서가 ‘매뉴얼 참고’로만 기재되어 해당 내용을 특정할 수 없습니다. 문서명·버전·장/절 또는 페이지와 해당 위치에서 확인 가능한 내용을 기재해 주시기 바랍니다. [관련 기준 1.1, 2.1, 2.3]`

### 구현방식 재분류

`기능 F__는 신청서에서 D(온디바이스 AI)로 선택되었으나, 제출 구조도상 장치 내부 모델 실행 없이 클라우드 API를 호출하는 것으로 확인됩니다. 실제 실행구조를 기준으로 구현방식을 C 또는 E로 재검토하고, 외부 API 정보와 모델별 입출력 흐름을 보완해 주시기 바랍니다.`

## 3. A 유형 보완 문구

### 학습데이터

`자체 학습 모델의 학습데이터가 ‘자체 데이터 10만 건’으로만 기재되어 출처·유형·라벨·보유형태를 확인할 수 없습니다. 데이터 수집기간, 데이터 유형 및 규모, 라벨 정의·검수방법, 학습/검증/시험 분리, 이용권한을 포함하여 보완해 주시기 바랍니다. [관련 기준 1.2]`

### 개발환경·알고리즘

`개발환경이 ‘Python 딥러닝’으로만 기재되어 모델 구조와 실제 배포 산출물을 식별하기 어렵습니다. 개발언어·프레임워크 버전, 모델/알고리즘 명칭, 최종 모델 파일 형식·파일명 또는 해시를 보완해 주시기 바랍니다. [관련 기준 1.1, 1.2]`

### 학습 주장과 실제 불일치

`신청자료에는 자체 학습으로 기재되어 있으나 현장 확인 결과 운영환경은 외부 API 응답만 사용하고 자체 모델 파일 또는 학습 산출물이 확인되지 않았습니다. 자체 학습 모델의 실제 적용 근거를 제시하거나 구현방식과 신청 기능명세를 실제 구조에 맞게 정정해 주시기 바랍니다.`

## 4. B 유형 보완 문구

### Base Model 불명확

`Base Model이 ‘BERT’로만 기재되어 정확한 모델을 식별할 수 없습니다. 제공자 또는 저장소, 전체 모델명, 버전·리비전·커밋과 실제 운영 체크포인트의 관계를 보완해 주시기 바랍니다. [관련 기준 1.1, 1.2]`

### 튜닝 없음/불명확

`튜닝 방법이 ‘사내 데이터 학습’으로만 기재되어 원본 모델과 변경된 모델을 구분하기 어렵습니다. 전체 파인튜닝·전이학습·LoRA 등 적용방법, 학습 대상 계층, 주요 조건과 최종 체크포인트 선택기준을 보완해 주시기 바랍니다. 튜닝하지 않은 경우에는 ‘튜닝 없음, 원본 가중치 사용’으로 명확히 기재해 주시기 바랍니다.`

## 5. C 유형 보완 문구

### 외부 API 식별

`외부 API 정보가 ‘ChatGPT 사용’으로만 기재되어 제공자·모델·배포 버전을 식별할 수 없습니다. 서비스 제공자, API/모델명, 버전 또는 배포명, 제품 내부 호출 모듈, 응답의 제품 기능 반영방법을 보완해 주시기 바랍니다. [관련 기준 1.1, 3.1]`

### 호출 로그

`현장 시연 시 기능은 실행되었으나 해당 시각의 외부 API 호출 로그·request ID·사용량 증가가 확인되지 않았습니다. 시연 기능과 연결되는 마스킹 호출 로그 또는 제공자 대시보드 사용내역을 제출해 주시기 바랍니다. [관련 기준 3.1]`

### 단순 중개 위험

`외부 AI 응답이 제품 화면에 그대로 표시되고, 제품 데이터 전처리·업무기준 결합·응답 검증·후속 처리 관계가 확인되지 않았습니다. 외부 AI 출력이 신청 제품의 주요 기능과 기술적으로 결합되는 처리 흐름과 관련 자료를 보완해 주시기 바랍니다. [관련 기준 2.1, 2.2]`

## 6. D 유형 보완 문구

### 타겟 환경

`타겟 환경이 ‘Jetson/Linux’로만 기재되어 심사대상 장치와 추론환경을 특정할 수 없습니다. 제품 모델명, 보드·SoC·메모리, OS/펌웨어 버전, 실제 모델 엔진 파일과 추론런타임 버전을 보완해 주시기 바랍니다. [관련 기준 1.2, 2.1]`

### 로컬 추론 미확인

`온디바이스 AI로 신청하였으나 네트워크 차단 상태에서 기능이 동작하지 않았고 장치 내부 모델 파일·로딩 로그가 확인되지 않았습니다. 로컬 추론 여부를 확인할 수 있는 모델 엔진, 런타임 로그, 오프라인 시연을 보완해 주시기 바랍니다.`

## 7. E 유형 보완 문구

### 구성만 나열

`혼합 구성 설명이 모델명 나열에 그쳐 모델별 역할과 데이터 연결관계를 확인하기 어렵습니다. 각 모델의 실행 위치, 입력, 출력, 다음 처리단계와 최종 기능 결정방식을 순서대로 보완해 주시기 바랍니다. [관련 기준 1.3, 2.1]`

### 일부 모델만 확인

`혼합형 기능 중 엣지 객체탐지 모델은 확인되었으나 외부 VLM 호출 및 두 결과의 결합 로직이 확인되지 않았습니다. 외부 호출 로그와 event ID 기준의 중간결과 연결, 최종 판정 로직을 보완해 주시기 바랍니다. [관련 기준 2.1, 2.3, 3.1]`

- 신청기업이 보완자료를 제출해 KOSA가 같은 접수번호로 엑셀을 다시 넘기더라도, **시트 값은 자동으로 최신화되지 않습니다.** 기능명세·구조도파일명 등이 바뀌었다면 관리자가 스프레드시트를 직접 수정합니다.
- 따라서 “접수 이후 신청서 내용이 바뀐 적이 있는가”는 시스템이 자동으로 알려주지 않으며, 접수대장의 `비고`나 파싱로그를 심사원이 직접 확인해야 합니다.
- §00 §2에서 지적된 “README는 갱신, 코드는 건너뜀”이라는 불일치는 코드 쪽이 아니라 **README·`입력폼_명세서.md` 서술이 틀렸던 것**으로 확인되어 두 문서를 코드 동작(건너뜀)에 맞게 수정했습니다.

### 잘못 등록한 접수 건 삭제

자동 삭제 기능은 제공하지 않습니다. 관리자가 동일한 `접수번호`를 기준으로 **접수대장·일정관리·인공지능제품모델·인공지능기능상세**의 관련 행을 모두 직접 삭제합니다. `파싱로그`는 작업 이력으로 유지하며, Drive의 접수번호 폴더는 필요 여부를 확인한 뒤 별도로 정리합니다.

## 2. 마감일이 두 곳에서 서로 다르게 계산됩니다

| 위치 | 계산식 | 기산일 | 코드 위치 |
|---|---|---|---|
| 접수대장 `심사마감일` (+ Chat/이메일 알림 문구) | 기산일 + `CONFIG.기본심사기간`(기본값 14일) | 파싱을 실행한 시각(오늘) | `_Sheets에등록` |
| 일정관리 `마감예정일` | 신청일 + 15 WD (주말·대한민국 공휴일 제외) | 신청일(신청서 원본값) | `_일정관리행추가` |

이 둘은 서로 참조하지 않는 **완전히 별개의 계산**입니다. 신청일과 심사접수일(파싱일) 사이에 며칠이라도 차이가 있으면 두 값이 어긋납니다.

**심사원에게 주는 의미**

- 「인공지능 제품·서비스 확인 절차 운영에 관한 고시」상 처리기한의 법적 기산점은 **신청일**이므로(README·본 가이드 00장 기준), 실무 판단에는 **일정관리의 `마감예정일`(신청일+15 WD, 주말·공휴일 제외)** 을 우선 사용하십시오.
- 접수대장의 `심사마감일`과 Chat/이메일 알림에 찍히는 마감일은 `CONFIG.기본심사기간`(14일, 심사접수일 기준) 값으로, 법적 마감일과 다를 수 있는 **내부 참고값**입니다. 두 값이 다르게 보여도 시스템 오류가 아닙니다.
- `CONFIG.기본심사기간`을 바꾸더라도 일정관리의 마감예정일 수식(15 WD 고정)에는 영향이 없습니다. 두 값은 기산점과 용도가 서로 다릅니다.
- 일정관리의 `보완요청일`은 내부 심사원이 직접 입력하고 `연장마감일`은 보완요청일+30 WD로 자동 계산하며, 접수대장과 동기화하지 않습니다. 보완요청 전에는 `마감예정일`, 보완요청 후에는 `연장마감일`이 만료되면 해당 행을 빨간색으로 표시합니다.
- `특이사항`은 담당심사원 오른쪽의 내부 협업용 자유 입력 컬럼입니다. 모든 사용자가 특이사항 또는 의견을 작성할 수 있으며 접수대장·보고서와 자동 동기화되지 않습니다. Google Sheets의 시트 및 범위 보호 설정에서 이 컬럼만 공용 편집 범위로 둡니다.

## 3. 기능번호는 “F01” 형식이 아니어도 됩니다

02장 §2.1 표의 “적정 예: `F01`”은 내부적으로 읽기 좋은 표기 관례를 제안한 것이며, 실제 KOSA 엑셀 원본 데이터(예: `샘플_신청데이터_10건.xlsx`)의 기능번호는 대부분 `1`, `2`, `3`처럼 **단순 정수**로 들어옵니다. 이 자체는 미흡 사유가 아닙니다.

같은 표의 “미흡 예: 모든 기능을 `1`로 기재”가 실제로 지적하는 문제는 **번호 형식**이 아니라 **한 접수 건 안에서 서로 다른 기능에 같은 번호를 중복 부여하는 것**입니다. 심사원은 번호가 `F01`인지 `1`인지가 아니라, *기능 수만큼 번호가 중복 없이 부여되었는지*를 확인하십시오.

## 4. ‘조건부 적합’은 아직 정책이 확정되지 않았습니다

접수대장 `종합판정` 드롭다운에는 `적합 / 조건부 적합 / 부적합` 세 값이 있지만, 04장 §10 “종합판정 로직”은 적합/보완 진행/부적합 세 갈래만 정의하고 있어 **‘조건부 적합’이 어느 경우에 해당하는지 본 가이드에 명시되어 있지 않습니다.**

정책이 확정되기 전까지는 다음과 같이 운용하는 것을 권고합니다.

- ‘조건부 적합’은 확인기관(KOSA)과 사전 협의 없이 최종 처분값으로 확정하지 않는다.
- 애매한 건은 04장의 “보완 진행”으로 우선 분류하고, 종합의견에 조건부 요소를 구체적으로 기재한다.

## 5. 보고서에는 “오픈소스 모델”이라는 짧은 이름이 그대로 노출됩니다

본 가이드 §3.2와 05·06장 예시는 구현방식 B를 설명할 때 `오픈소스·사전학습 모델 활용`처럼 풀어서 씁니다. 그러나 `Code.js`의 `_구현방식레이블` 함수가 실제 보고서(Google Docs)에 출력하는 문구는 짧은 코드 레이블 `오픈소스 모델`입니다(A~E 전부 코드 레이블 그대로 출력됨: 자체 학습 모델 / 오픈소스 모델 / 외부 AI API 연동 / 온디바이스 AI / 혼합형 AI 구성).

심사원이 작성한 종합의견 문구와 시스템이 자동 생성한 보고서 표의 구현방식 명칭이 다르게 보이더라도, 같은 항목을 가리키는 것이므로 오류가 아닙니다. 다만 보고서를 신청기업·KOSA에 전달할 때 용어 차이로 혼동이 없도록, 필요하면 종합의견에 코드와 함께(A~E) 표기하는 것을 권고합니다.

## 6. 필드 필수성은 문서마다 표현 수준이 다릅니다

`입력폼_명세서.md`는 접수현황·제품모델·인공지능기능상세 각 항목을 “필수 / 조건부 필수 / 선택” 3단계로만 표시합니다. 반면 02장 §10 “구현방식별 필수성 매트릭스”는 같은 항목이라도 구현방식(A~E)에 따라 ● / ○ / — 로 더 세분화되어 있습니다(예: `연산자원요약`은 A·B·D·E에서 ●이지만 C에서는 ○).

두 문서가 상충하는 것은 아니며, **`입력폼_명세서.md`는 “KOSA 입력폼 관점의 필수 여부”**, **02장 매트릭스는 “심사 판단 관점에서 실질적으로 중요한 정도”** 를 나타냅니다. 필드별 심사 중요도를 판단할 때는 02장의 매트릭스를 우선 참고하십시오.
